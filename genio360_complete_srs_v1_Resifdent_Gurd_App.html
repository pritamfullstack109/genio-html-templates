Architecture & Cross-Cutting Standards

Technical Architecture Overview

The contracts every module inherits — Flutter, NestJS, Azure

All requests from Flutter apps traverse Azure API Management → NestJS (AKS) → Azure data stores. The backend is the single point of authorization: it derives the caller's role and condo scope from the JWT on every request and enforces both RBAC and module-visibility flags server-side.

REQUEST PATH Flutter App (Resident / Guard) ──HTTPS + Bearer JWT──▶ NestJS API Gateway │ ▼ Guard chain: JwtAuthGuard → SessionGuard → RolesGuard → ModuleEnabledGuard → CompanyGuard │ (AuditInterceptor logs every write) ▼ Domain services (Auth / Booking / Package / Guard / …) ──WHERE companyId = scope──▶ Data stores DATA STORES SQL transactional / relational entities (users, units, bookings, requests) Azure Blob media / photos / documents (SAS URLs) Azure Service Bus async jobs (email, notifications, exports) Azure Key Vault all secrets — JWT signing keys, AES encryption keys, OTP secrets Redis sessions · JWT denylist · FCM token cache · module flags

Cross-Cutting Standards (every module inherits)

CC-01 JWT Auth → JwtAuthGuard on every route  ·  CC-02 Scope → companyId/condoId filter on all queries  ·  CC-03 Soft delete → is_deleted / status flags  ·  CC-04 Encryption → PII columns + Key Vault  ·  CC-05 Validation → ValidationPipe + DTOs  ·  CC-06 Audit → AuditInterceptor → audit log  ·  CC-07 Notifications → FCM push via Azure Service Bus worker

Standard Error Codes

400 validation · 401 auth/OTP · 403 out-of-scope / RBAC · 404 not found · 409 conflict · 422 semantically invalid · 429 rate-limit · 500 internal. Every response envelope contains a correlationId.

Module 01

Authentication Overview

Login · Sign Up · Forgot Password · Logout — shared auth stack for Resident & Guard apps

Roles: Resident · Security Guard
NestJS module: AuthModule
Services: AuthService · OtpService · SessionService · TokenService

Objective

Secures platform access for all user types. Validates identities, issues JWT session tokens, enforces condo-level policies (2FA, SAML, T&C, password complexity), and routes authenticated users to the correct experience — Resident Home or Security Guard Shell.

Architectural Approach

Both Flutter apps share the auth_shared package. The Flutter app holds JWT in memory (access) and flutter_secure_storage (refresh). A Dio interceptor auto-calls POST /v1/auth/token/refresh on 401. RBAC claim in JWT routes to ResidentHome or GuardShell.

Module 02

Login

Email/password login with SAML SSO, 2FA OTP, condo/unit selection, and JWT session issuance

Role: Resident / Security Guard
Screen: LoginPage
Endpoint: POST /v1/auth/login

Objective

Authenticates residents and security guards using email + password (or SAML SSO), validates 2FA OTP when required, guides users through condo/unit selection, and issues session tokens for subsequent API access.

Architectural Approach

Flutter Dio client posts LoginDto. On success, JWT stored in memory + refresh in flutter_secure_storage. SSL pinning via dio_certificate_pinning. RBAC role claim in JWT determines routing: Resident → ResidentHome, Guard → GuardShell.

Module 05

Logout

User-initiated and force logout; session revocation; token cleanup

Role: Resident / Guard
Endpoint: POST /v1/auth/logout

Functional Requirements

  • FR-01User-initiated logout: confirmation dialog → POST /v1/auth/logout.
  • FR-02System revokes JWT + refresh token on server. Clears Redis session entry.
  • FR-03Flutter clears all tokens from flutter_secure_storage and Hive cache.
  • FR-04Navigation clears back stack → LoginPage. User cannot press Back to re-enter app.
  • FR-05Force logout: FCM push with force_logout payload or session status returns revoked → same cleanup flow.
  • FR-06Idle timeout: if forceLogoutInactivePeriod minutes elapsed without interaction → auto-logout.

Business Rules

All auth tokens removed locally on logout. Logout from one device does not affect sessions on other devices (per-device session model). Admin-sensitive screens poll session status on resume to detect force logout.

Module 20

Virtual Button

Remote door/gate open via MQTT — residents unlock building access from mobile

Role: Resident
Endpoint: POST /v1/doors/:id/remote-open

Objective

Allows authenticated residents to remotely open doors and gates via a virtual button that triggers MQTT commands to access control hardware.

Architectural Approach

Flutter button → POST /v1/doors/:id/remote-open → NestJS driver-mqtt → MQTT broker → hardware controller. WebSocket event door_opened confirms execution.

Functional Requirements

  • FR-01List of accessible doors/gates based on resident unit and permissions.
  • FR-02Large tappable virtual button per door. Confirmation dialog before sending open command.
  • FR-03WebSocket event door_opened confirms hardware execution with timestamp.
  • FR-04Cooldown timer prevents rapid repeated presses (configurable, e.g. 5-second lock).
  • FR-05Audit log entry created for every remote open event (resident ID, door ID, timestamp).

Business Rules

Virtual button available only within configured operating hours if time-restriction is enabled. Hardware command failures (MQTT timeout) shown as error with retry option.
