Architecture & Cross-Cutting Standards

Technical Architecture Overview

Sections 1–6 — the contracts every module inherits

The Installer Portal is one role-rendering of the global React Admin SPA. All requests traverse Azure API Management → the regional AKS cluster (NestJS) → regional data stores. The backend is the single point of authorization: it derives the caller's tier and scope from the JWT on every request and enforces both RBAC and the funnel server-side.

REQUEST PATH Installer (React SPA) ──HTTPS + Bearer JWT──▶ Azure API Management (routes by cluster_id) │ ▼ NestJS API Gateway / BFF ──▶ Guard chain: JwtAuthGuard → SessionGuard → RolesGuard → ScopeGuard → ModuleEnabledGuard │ (AuditInterceptor logs writes) ▼ Domain services (Property / Unit / Staff / …) ──WHERE installer_id = scope──▶ Data stores DATA STORES MySQL (Azure) transactional/relational entities Cosmos DB high-write logs / audit / forms / events Redis sessions (single) · JWT denylist · funnel module flags Azure Blob media / CSV / exports Azure Key Vault all secrets, by alias (never in config/logs)

Cross-Cutting Standards (every module inherits)

CC-01 Funnel → ModuleEnabledGuard + Redis flags  ·  CC-02 Scope → ScopeGuard installer_id  ·  CC-03 Soft delete → status/deleted_at  ·  CC-04 Encryption → PII columns + Key Vault  ·  CC-05 Validation → ValidationPipe + DTOs  ·  CC-06 Audit → AuditInterceptor → System Config trail  ·  CC-07 Onboarding email → async queue  ·  Localization → en/zh-SG/vi.

Standard Error Codes

400 validation · 401 auth/OTP · 403 out-of-scope / funnel · 404 not found · 409 conflict · 422 semantically invalid · 429 rate-limit · 500 internal. Every response uses one envelope with a correlation_id.

Module 01

Authentication

Login, MFA, Single-Session, Forgot/Reset/Set Password, Logout

Role: Installer (Web Portal)
NestJS module: AuthModule
Services: AuthService · OtpService · SessionService · TokenService

Objective

Provides secure authentication, multi-factor verification, single-device session enforcement, password recovery/activation (with mandatory Terms & Conditions acceptance), and issuance of the role-and-scope-bearing JWT every other module relies on for authorization.

Architectural Approach

The frontend is a React SPA. Credential validation is delegated to the Shared Auth DB; on success AuthModule issues a signed JWT access token + refresh token. React holds the access token in memory (refresh token in an httpOnly cookie), decodes perms + scope (installer_id, cluster_id), and an Axios interceptor attaches the Bearer token and auto-calls /auth/refresh on a 401.

Scope

Credential validationMFA / OTPJWT issuanceSingle-session (1 device)Password recoveryFirst-time activation + T&CLockout / enumeration prevention

Login & MFA Flow

1
User submits credentials
React captures Email/Password via a controlled form and posts over HTTPS.
POST /api/v1/auth/login {email, password} → delegated to the Shared Auth DB. Invalid → 401 generic (no enumeration).
2
MFA / OTP challenge
On valid credentials, tokens are not issued until OTP is verified.
Backend returns 200 {mfa_required, challengeId}; SPA calls POST /api/v1/auth/verify-otp. OTP single-use & time-limited; invalid → 401.
3
JWT issued + single session written
The role-and-scope token is issued and the session registered in Redis.
JWT: sub, role, tier, cluster_id, distributor_id, installer_id, perms[], sid. Redis session:{userId}→sid overwrites & denylists prior. Single-session
4
Scoped redirect + RBAC render
ProtectedRoute renders only authorized routes, then redirects to the Dashboard.
SPA hides unauthorized UI; backend stays authoritative. Data scoped to installer_id.
5
Token refresh & lockout
SPA monitors exp; interceptor auto-refreshes or forces logout.
Redis failed-attempt counter → 401 "Account locked". 5 fails / 15 min

Redis session / denylist / lockout. DTOs: LoginDto, VerifyOtpDto, ResetPasswordDto, SetPasswordDto{acceptTnc}.

Module 06

Property (Condo) Management

Create / View / Search / Export — the core onboarding entity

Role: Installer (Web Portal)
NestJS module: PropertyModule
Services: PropertyService · PropertyConfigService · PropertyExportService

Objective

Enables the Installer to create a property and its full configuration in one flow, list/search/filter, and export — the core onboarding entity for the platform.

Architectural Approach

React renders a multi-section creation form; branding assets upload to Azure Blob (the API stores URLs). Creation runs inside a single MySQL transaction across the condo + child config tables; on commit an onboarding email is enqueued and an audit entry written. The selectable module set is constrained server-side to the Distributor-permitted subset (the funnel). Data resides in the Installer's cluster_id region.

session:{userId}→sid (= refresh-token TTL) · token denylist (= remaining access TTL) · perms:{userId} (15 min) · moduleFlags:{condoId} funnel (10 min) · dashboard counters (1–5 min).

Query performance

Cosmos queries must be partition-scoped + date-bounded; cross-partition fan-out prohibited on interactive paths (use async export). Partition keys: system_config_audit=installer_id, operational_audit/anpr_event=condo_id, people_count_event=condo_id/camera_id, notification_queue=campaign_id.
