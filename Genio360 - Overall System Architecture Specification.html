Executive Summary

Modernization Strategy at a Glance

From a single-tenant Singapore monolith to a white-label, multi-region governance platform

The legacy iPlusLiving system is a PHP/Yii2 monolith (mobile clients in Objective-C / Java) scoped only by company_id / condo_id, hosted single-region. Genio360 re-platforms this into a four-tier, geo-partitioned, multi-tenant ecosystem, deployed end-to-end on Microsoft Azure. Web portals are built in React; mobile apps are built in Flutter; both clients talk to the same NestJS (TypeScript) backend running on Azure. Every capability above the property layer (HQ governance, Distributor channel management, the funnel) is NEW; property and resident features are migrated As-Is with optimization.

🌐

White-Label, Multi-Region

One codebase serves Europe, Asia, and Middle East clusters; each region is a fully isolated Azure Resource Group. Data never crosses regional boundaries.

🪜

Hierarchical Governance

HQ → Distributor → Installer → Property/Resident. Each tier only acts on what it owns; upstream may observe downstream but never operate it.

🚰

Progressive Funnel

Module availability cascades downward — each tier can only enable a subset of what its parent permitted, enforced server-side at the API.

📦

Batched, Phased Rollout

Genio360 Lite (priority modules) goes live first; the full platform follows. Properties migrate to the new MeetMe Stream SDK in phased batches.

Tech-Stack Modernization (one cloud: Azure)

Web — React 19 SPA (Vite) on Azure Static Web Apps + CDN.  Mobile — Flutter (iOS + Android) calling the same API.  Backend — NestJS (TypeScript) microservices on Azure Kubernetes Service (AKS), fronted by Azure API Management.  Data — Azure Database for MySQL/PostgreSQL + Azure Cosmos DB.  Platform — Azure Cache for Redis, Blob Storage, Key Vault, Service Bus, Web PubSub, Functions, App Insights. Push remains on Firebase Cloud Messaging (device delivery only). No AWS dependencies.

Section 2 · Topology

Global Shared Services vs. Regional Infrastructure

A globally-shared control band over independent, data-isolated regional clusters

The platform separates globally-shared control services (authentication, API routing, secrets, the one React Admin SPA) from regional execution clusters that hold all tenant data. A request authenticates against the global Shared Auth DB, receives a JWT carrying cluster_id / distributor_id / role, and Azure API Management routes it to the correct regional cluster. Tenant data is resident in its region only and never crosses boundaries.

Routing & Isolation Contract

APIM resolves the target cluster from the token's cluster_id; there is no cross-region data path. Retention/purge jobs run independently per cluster. The EU cluster enforces GDPR; Asia and Middle East adopt equivalent policies. Multiple distributors may share a region but are mutually isolated by entity_id.

Standard Response Envelope

Every API response uses one envelope with a correlation_id. Error codes: 400 validation · 401 unauthenticated · 403 out-of-scope / funnel · 404 not found · 409 conflict · 422 semantically invalid · 429 rate-limit · 500 internal.

Shared Control Services — Resilience

The Shared Auth DB, APIM, and Front Door are global dependencies. Their failure would affect all regions, so each is designed for high availability with explicit failover behavior.

Shared Auth DB is the Global SPOF

Even with HA, a global Shared Auth DB outage is the highest-impact failure. Mitigations: JWT JWKS cached at APIM, Redis session store per region, and a documented degraded mode where existing sessions continue while new logins are blocked. Login retry logic must be graceful.

Section 3 · Governance Model

The 4-Tier Hierarchical Governance Model

Ownership-based access — each layer acts only on what it owns; no lateral or upward operation

Genio360 operates across four administrative levels. The escalation path is strictly User → Property → Installer → Distributor → HQ (direct User→HQ is not permitted). Upstream layers may observe downstream data for oversight, but cannot operate it; downstream layers have no visibility upward or laterally.

🏛️

Tier 1 (L1) — HQ / FAP

Strategy & governance authority (Fermax Asia Pacific).

  • Provision distributors under region clusters
  • Define the global module catalogue (top of funnel)
  • Set global roles, security & data-retention policies
  • System-wide view-only oversight — no day-to-day ops
🤝

Tier 2 (L2) — Distributor

Channel layer; bridge between HQ and execution.

  • Scoped to assigned country/countries (entity_id)
  • Allocate a subset of HQ-permitted modules to installers
  • Onboard & manage installation companies
  • Read-only aggregate analytics; no upward/lateral view
🔧

Tier 3 (L3) — Installation Company

Technical execution & project delivery.

  • Create & provision properties (units, panels, keys)
  • Call-divert provisioning & system integrations setup
  • Enable a subset of distributor-allowed modules per project
  • Assign Property Managers; review deployment/system logs
🏢

Tier 4 (L4) — Property Mgmt & Residents

Daily building operations & end-user features.

  • PM: configuration, approvals, access-control events, reports
  • Hardware integrations (intercom, ANPR, FR, parking)
  • Resident: bookings, QR/face entry, wall, wallet, feedback
  • Residents strictly isolated to their own unit
⬇️

Downward Permission Flow

Constraints inherit top-down. A tier can never grant more than it was granted — the funnel ceiling propagates HQ → Distributor → Installer → Condo.

🚨

Escalation (Upward Only)

Issues travel User → Property → Installer → Distributor → HQ. HQ may receive explicit, time-limited, read-only temporary access — never standing operational control.

Section 4 · Funnel

The Progressive Module Enablement Funnel

Module availability cascades downward — a tier can only enable what its parent permitted

The "funnel" is the platform's core multi-tenant safety mechanism. HQ defines the master module catalogue (the ceiling). Each tier narrows that set for the tier below. The constraint is enforced server-side at the API — never merely hidden in the UI — using a NestJS ModuleEnabledGuard backed by fast-lookup Redis flags, so a tier can never exceed its parent's ceiling and cannot leak data laterally or upward.

1
HQ Ceiling — Global Catalogue
HQ (L1) defines the full module set and the subset each Distributor MAY use.
Top of funnel: moduleCatalogue[]. HQ enables a per-distributor allow-list; this is the absolute maximum any descendant can ever see.
2
Distributor Assignment
Distributor (L2) enables a subset of HQ's allow-list for each Installer.
Constraint: installerModules[] ⊆ distributorAllowed[]. Exceeding HQ's ceiling → 403.
3
Installer Project Provisioning
Installer (L3) enables a subset of distributor-allowed modules per property.
On POST /api/v1/properties: moduleIds[] ⊆ Distributor-allowed (else 403). Modules bind to the property at provisioning time.
4
Condo Feature Toggle
Property Management (L4) toggles features within the installer-provisioned set.
PM can switch on/off only modules already provisioned for the property; it has no configure-upward option.
API-LEVEL ENFORCEMENT Request ──▶ JwtAuthGuard ──▶ SessionGuard ──▶ RolesGuard ──▶ ScopeGuard ──▶ ModuleEnabledGuard ──▶ Handler │ │ (WHERE *_id = caller scope)─────┘ │ Redis: module-flags:{cluster_id}:{tenant_id} ────────┘ (cache miss → DB → backfill) AuditInterceptor logs every config write to the System Config Audit Trail

Why API-Level (not UI-Level)

Hiding a button is not security. ModuleEnabledGuard rejects any request for a module not enabled for the caller's tenant with 403, regardless of how the request was crafted. This prevents lateral (peer tenant) and upward (parent tier) data leaks even if the SPA is bypassed.

Module-Flags Cache Invalidation

Funnel flags are cached in Redis with a TTL. When a parent tier changes a module allowance, the config-write API publishes an invalidation event to a Redis pub/sub channel; each regional gateway refreshes its cache and evicts the stale key. A fallback background job reconciles cache vs. DB every 60 seconds.

INVALIDATION FLOW HQ/Distributor/Installer changes module allowance → ConfigService writes DB → Publish module-flags:invalidate:{cluster_id}:{tenant_id} to Redis pub/sub → All NestJS gateways delete the key → Next request: cache miss → DB read → backfill Redis → Reconciliation job (60s) ensures no stale keys remain

Resilience Patterns for External Integrations

Every external call uses a resilient client configured per integration criticality. Timeouts and retries are tuned to the vendor's observed behavior; circuit breakers prevent cascading failures; bulkheads isolate vendor failures from the main request path.

Webhook Handling

Vendor webhooks (Stripe payment events, Hikvision callbacks) are received by a dedicated ingress path, validated by signature, and immediately persisted to an idempotent queue in Service Bus. Processing is asynchronous to avoid blocking the vendor and to provide replay capability.

WEBHOOK INGEST Vendor POST webhook → Front Door → WAF → APIM → WebhookController (AKS) → Validate signature + idempotency key → Persist raw payload to Cosmos → Enqueue to Service Bus topic → Azure Function / consumer processes business logic → Ack 200 to vendor (or 202 if async) → Failed validation returns 400; retries handled by vendor
Section 6 · Engineering "How"

Legacy Optimization & Performance Fixes

Structural data migrations that resolve the legacy system's deadlocks, slow reads, and instance-bound load

The Scope of Work catalogues recurring legacy pain points — MySQL deadlocks under concurrency, slow ANPR barrier responses, instance-bound WebSocket load, and notification fan-out delays. Each is resolved by moving the offending workload off the relational hot path onto a fit-for-purpose Azure store (Azure Cosmos DB for high-write, Azure Cache for Redis for atomic counters/cache, Azure Blob for files, Azure Functions for fan-out).

1
File Storage Migration
High-write attachment rows in MySQL caused read/write contention.
LegacyAttachment links + metadata in MySQL attachment table; binaries on AWS S3. Frequent add/update/delete causes performance issues.
ProposedBinaries to Azure Blob Storage (private + SAS); document links/metadata to Cosmos DB. MySQL is relieved of the high-write churn.
2
eAGM Voting / Poll Deadlocks
Simultaneous poll submissions deadlocked MySQL and slowed other modules.
LegacyPolls stored in MySQL with complex config; concurrent submissions periodically deadlock the server and stall unrelated modules.
ProposedLive tallies in atomic Azure Cache for Redis count keys (INCR); async persistence to Azure Cosmos DB. Concurrency is lock-free; relational store is no longer the hot path.
3
Notification Engine & WebSockets
50+ trigger points and an on-instance socket created peak-time load on Web 1.
LegacySelf-hosted socket on a single web instance (refresh for Android & Web); 50+ notification triggers with many conditions add load.
ProposedManaged Azure Web PubSub for fan-out; notification logic moved to serverless Azure Functions with Cosmos DB storage — decoupled from the request path.
4
ANPR Logging Overhead
Large vehicle in/out tables in MySQL slowed barrier workflow and caused deadlocks.
LegacyVehicle In/Out/Event in MySQL grows huge; ANPR needs fast response but the shared table deadlocks (check-in time vs previous-entry reads).
ProposedHigh-throughput entry/exit history to an isolated Azure Cosmos DB partition pattern; blacklisted-vehicle lookups served from Azure Cache for Redis. Barrier latency and cross-module slowness are eliminated.
5
MQTT Event History & FR Sync
MQTT events flooded MySQL; FR face-load rules hammered the DB.
LegacyMQTT generates many event rows in MySQL; searching them causes downtime elsewhere. FR photo-sync checks many DB rules per panel.
ProposedHistorical MQTT events to Azure Cosmos DB; FR sync rules cached in Azure Cache for Redis so faces load fast without high DB CPU. The Azure Event Grid MQTT broker auto-scales for peaks.

Common Pattern

Keep Azure Database for MySQL/PostgreSQL for transactional truth; offload high-write, append-only, or ephemeral workloads (logs, events, tallies, history, sessions, flags) to Azure Cosmos DB and Azure Cache for Redis; put binaries in Azure Blob; run fan-out on Azure Functions. This removes deadlocks and decouples peak load from the request path — all within Azure.

Migration Strategy

The legacy PHP/Yii2 monolith is migrated using a Strangler Fig pattern: functionality is extracted bounded-context by bounded-context, wrapped behind the NestJS API, and switched over via feature flags. High-risk modules (payments, access-control) are dual-written during cut-over so rollback is immediate.

1
Wrap
Add a NestJS facade in front of the legacy PHP module for reads and writes.
2
Migrate
Move business logic and data to the new Azure-backed service; dual-write to legacy for safety.
3
Switch
Feature flag routes traffic to the new service; monitor SLOs for 48h before disabling legacy path.
4
Retire
Stop dual-write, archive legacy data, decommission PHP module.

Rollback Plan

Each module has a 5-minute rollback plan: toggle feature flag back to legacy, drain in-flight requests, and verify with synthetic checks. DB changes are backward-compatible (expand/contract) so app rollback never breaks schema.

Section 15 · Resilience

Data Ownership Principle

Every row/record carries cluster_id and tenant_id. All queries are scoped by these columns; the database layer has Row-Level Security (RLS) policies as a second line of defence. Soft-delete is implemented via is_deleted / status with a hard purge after retention.

ER & Partitioning Strategy

Relational ER is bounded-context split: auth, tenant, property, access-control, payments. Cosmos containers are partitioned by cluster_id to keep tenant data physically co-located and avoid cross-partition hot spots. Sharding is not needed initially — horizontal scaling is handled by Cosmos throughput and read replicas.

Data Classification & PII Data Flow

All data is classified into four tiers. PII and payment data are Restricted; access logs and audit trails are Confidential; configuration and marketing data are Internal; static help content is Public.

Non-Production Data & Cosmos Replication

Production data never flows to dev/uat. Instead, a masked/anonymized subset is generated using synthetic data or tokenized production samples. Cosmos DB uses single-region write with multi-region read replicas for low-latency reads; multi-region writes are enabled only if a region requires local write latency and conflict resolution is acceptable.

Anonymization

Names are hashed; phone numbers are tokenized; face photos are replaced with synthetic images; NRICs are redacted. The process is run by an Azure Function that scans production backups and produces a sanitized dataset.

Cosmos Replication Model

Default: single-region write with 1–2 read replicas. Writes always go to the primary region to avoid conflicts. Reads can be served from the nearest replica if eventual consistency is acceptable. GDPR boundaries are respected: EU replicas are in EU regions only.

Section 18 · Decisions

Architecture Decision Records (ADRs)

Rationale for major design choices

ADR-001 — Why AKS instead of App Service / Container Apps

Context: Need to host 10+ bounded-context services at 1M-user scale with independent scaling, network policies, and a service mesh.

Alternatives: App Service (simpler but per-plan scaling limits and no service mesh), Container Apps (better for event-driven but less mature control over networking).

Decision: AKS. Private cluster, custom node pools, NetworkPolicy, HPA/VPA, mature service mesh, and enterprise support.

ADR-002 — Why Cosmos DB instead of MongoDB / DynamoDB

Context: High-write logs, audit, events, and forms must be globally distributed but Azure-native.

Alternatives: MongoDB Atlas (third-party, vendor ops), DynamoDB (AWS-only), PostgreSQL partitioning (complex at this scale).

Decision: Azure Cosmos DB for 99.999% SLA, turnkey global distribution, TTL, and unified Azure security.

ADR-003 — Why Azure API Management

Context: Single external gateway for web/mobile, routing to regional clusters, and enforcing auth/quota.

Alternatives: Application Gateway + custom auth layer (more code), Ambassador Edge Stack (operational burden).

Decision: APIM for built-in token validation, throttling, caching, routing, and centralized policy.

ADR-004 — Why Azure Cache for Redis

Context: Sessions, funnel flags, and FR sync rules need sub-millisecond, atomic access.

Alternatives: In-memory (no shared cache across pods), Memcached (no persistence/geo-replica).

Decision: Redis Premium for atomic operations, pub/sub, geo-replication, and high availability.

ADR-005 — Why Azure Functions

Context: Offload spiky, non-latency-sensitive work (emails, reports, FR sync, retention).

Alternatives: Always-on pods in AKS (waste), Logic Apps (less flexible for complex logic).

Decision: Functions for cost-efficient, scalable, queue-triggered processing that can scale to zero.

Additional ADRs

ADR-006 — Modular Monolith vs. Microservices

Context: The team could build a modular NestJS monolith on AKS instead of 10+ services.

Decision: Start with a modular monolith split into deployable bounded contexts (auth, tenant, property, access-control, payments, notifications). Each context can be scaled independently. This avoids premature microservices overhead while preserving future decomposition.

ADR-007 — Single React SPA vs. Separate Portals

Context: HQ, Distributor, Installer, and PM have different UIs but share patterns.

Decision: One React codebase deployed once, role-rendered by JWT. Reduces duplication and ensures a single CI/CD pipeline. The API guard chain enforces data boundaries.

ADR-008 — Flutter vs. Native Mobile

Context: iOS and Android apps need feature parity and fast iteration.

Decision: Flutter with two flavors (Resident, Security Guard). Single codebase, native plugin bridge for VoIP/FCM and hardware SDKs where needed.
