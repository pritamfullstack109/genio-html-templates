Architecture & Cross-Cutting Standards

Technical Architecture Overview

The contracts every module inherits

The Distributor Portal is one role-rendering of the global React Admin SPA. All requests traverse Azure API Management -> the regional AKS cluster (NestJS) -> data stores. The backend is the single point of authorization: it derives the caller's tier and scope from the JWT on every request and enforces RBAC and the funnel server-side. The Distributor's scope is its own entity only (distributor_id = entity_id); it can see its own installers/properties but has no upward or lateral visibility, and operational data inside modules is not exposed.

REQUEST PATH Distributor (React SPA) --HTTPS + Bearer JWT--> Azure API Management (routes by cluster_id) | v NestJS API Gateway / BFF ---> Guard chain: JwtAuthGuard -> SessionGuard -> RolesGuard (role=DISTRIBUTOR, tier=L2) -> ScopeGuard (WHERE distributor_id = own) -> ModuleEnabledGuard | (AuditInterceptor logs every config write) v Domain services (Installer / Module / Staff / Licence-view / Audit / ...) ---> Data stores SCOPE & FUNNEL HQ defines the module set the Distributor MAY use (top of funnel) Distributor enables a SUBSET of that set for each Installer (cannot exceed HQ's ceiling) Enforcement API-level (ModuleEnabledGuard + Redis flags), not just the UI DATA STORES (within the distributor's region cluster) MySQL (Azure) installers (installer_id), staff, roles, module-enablement, licence-allocation view Cosmos DB System Configuration Audit Trail (own scope; append-only, immutable) Redis sessions · module-flag (funnel) cache · perms cache Azure Blob report exports Data residency all data resident in the distributor's cluster_id region; mutually isolated by entity_id

Cross-Cutting Standards (every module inherits)

CC-01 Funnel - Distributor can only enable HQ-permitted modules for installers, enforced API-level  ·  CC-02 Scope - ScopeGuard distributor_id (own data only; cross-scope -> 403)  ·  CC-03 Soft delete - status (reactivatable; hard purge only via HQ retention job)  ·  CC-04 Encryption - PII columns + Key Vault  ·  CC-05 Validation - ValidationPipe + DTOs  ·  CC-06 Audit - AuditInterceptor -> System Config Audit Trail (own scope)  ·  CC-07 Onboarding email - async queue  ·  CC-08 Data residency - resident in the distributor's cluster_id.

Standard Error Codes

400 validation · 401 unauthenticated · 403 out-of-scope / forbidden / funnel · 404 not found · 409 conflict (e.g. duplicate email) · 422 semantically invalid · 429 rate-limit · 500 internal. Every response uses one envelope with a correlation_id.

Platform Context

Where the Distributor Sits

The channel tier between HQ governance and Installer execution

The Distributor (L2) is the bridge between HQ governance and operational execution. HQ (L1) creates and activates the Distributor; the Distributor then manages Installation Companies (L3) within its assigned country/countries, enables a subset of HQ-permitted modules for them, and supports their escalations. Multiple distributors may operate in the same country - each is a separate Distributor Entity (entity_id) and fully data-isolated.

The Four Tiers (Ownership-Based Access)

L1 HQ / FAP - governance, creates Distributors.  L2 Distributor - channel layer, manages installers within a country (this document; own entity_id).  L3 Installer - execution layer, onboards properties (own installer_id).  L4 Property Manager - site operations. Each layer acts only on what it owns; upstream may observe downstream but not operate it; downstream has no upward/lateral visibility.

Scope Note - Excluded from this revision (not in Global Requirements r5.0)

The following were present in earlier reference material but are not stated in Global Requirements r5.0, so they are excluded here pending a separate SOW/NFR sign-off: MFA / OTP + NFR-only login detail (single-session, lockout, forgot-password); Localization (en/zh-SG/vi); and Licence ALLOCATION to installers. The Global Requirements RBAC matrix grants the Distributor only "Licenses (MeetMe) - View own + modify password & validity" - allocation-to-installers is an SOW 4.4 extension beyond the matrix (see Open Items / Module 09).

Open Items (to confirm with FAP)

OI-1 Entity vs Region: cluster_id = region, entity_id = distributor (L2).  OI-2 Licence allocation: SOW 4.4 "allocate to installers" extends the Global Req RBAC matrix ("View own + modify password/validity") - confirm whether the authoritative matrix is updated; if so, add an allocation capability to Module 09.  OI-3 Escalated/temporary access mechanics to be confirmed in discovery.  OI-4 final role taxonomy / 66-module catalogue / funnel mapping.

Module 01

Authentication

Shared Auth DB credential validation + JWT issuance

Role: Distributor (Web Portal)
NestJS module: AuthModule
Services: AuthService · TokenService

Objective

Authenticate the Distributor user against the global Shared Auth DB and issue the role-and-scope JWT. (Per Global Req: the Shared Auth DB manages login credentials and JWT token generation; regional assignment - cluster_id, distributor_id, role - is resolved at login and embedded in the token.)

Architectural Approach

Credential validation is delegated to the Shared Auth DB; on success a signed JWT is issued carrying role + scope. Azure API Management routes the request to the distributor's cluster by cluster_id; the NestJS backend re-reads role/ids from the JWT on every request and scopes all queries to distributor_id.

Scope

Credential validation (Shared Auth DB)JWT issuanceScope resolution (distributor_id)Server-side RBAC render

Login Flow

1
Submit credentials
React posts Email/Password over HTTPS.
POST /api/v1/auth/login -> validated against the Shared Auth DB. Invalid -> 401.
2
JWT issued
Signed token carries role + scope.
JWT: sub, role=DISTRIBUTOR, tier=L2, cluster_id, distributor_id. APIM routes by cluster_id.
3
Scoped RBAC render
ProtectedRoute renders only Distributor routes; the backend re-validates the JWT on every request and scopes data to distributor_id.

Endpoints & Rules

POST /api/v1/auth/login -> Shared Auth DB validation, JWT issuance. Backend re-reads role/ids from the JWT on every request (single point of authorization). All authentication events written to the audit trail (Login History).

Persistence

Credentials/registry in the Shared Auth DB; session context in Redis. (MFA/OTP, single-session, lockout, forgot-password excluded - not in Global Req.)

Module 02

Distributor Dashboard NEW MODULE

Own-scope KPIs, read-only

Role: Distributor (Web Portal)
NestJS module: DashboardModule
Services: DashboardService

Objective

A role-scoped landing screen summarizing the Distributor's own scope: number of installation companies, properties, and aggregate user count (Global Req "Distributors - Visibility: Number of Installation Companies + Installers; Properties; Aggregate user count within the distributor's assigned scope").

React renders KPI cards from a single GET /dashboard/kpis, computed from aggregate read models scoped to distributor_id. Personal user data is never exposed upward (aggregate counts only). Note: the HQ "Global Dashboard" is No Access for L2; this is the distributor's own scoped view.

Module 04

Distributor Staff & Internal Roles

Create internal staff accounts and define/assign roles within scope

Role: Distributor (Web Portal)
NestJS module: StaffModule
Services: StaffService · RoleService

Objective

Create and manage the Distributor's own internal staff accounts and the internal roles assigned to them, within the distributor's scope (Global Req Process Flow: "Distributor creates internal staff roles").

Architectural Approach

Scoped CRUD over distributor staff and roles. Roles are defined within the distributor's scope; staff are assigned a role; new staff receive an activation email (async queue). All access stays bounded to distributor_id.

Access Model - full vs limited (role-driven)

A staff member's access is exactly what their assigned role permits - full or limited (least-privilege). Example: a staff with an installer-onboarding role can create installers; a "Viewer" role cannot - only reads. Every action is attributable in the audit trail.

Scope

CRUD staff (internal)Define / assign internal rolesFull or limited access (role-driven)Activation emailSoft delete

Endpoints & Rules

GET/POST/PUT/DELETE /api/v1/staff (paginated/search; email unique) · GET/POST/PUT /api/v1/roles (within scope) · PUT /api/v1/roles/{id}/permissions (evicts perms cache). Delete soft; all staff/role actions audited.

Persistence

MySQL staff_user (PII encrypted), role (distributor scope), role_permission.

Module 05

Installer Management

Create / Activate / Suspend / Soft-delete / List - the core entity

Role: Distributor (Web Portal)
NestJS module: InstallerModule
Services: InstallerService · InstallerOnboardingService

Objective

Create an Installation Company (Installer) with its primary admin and module selection, within the distributor's assigned country/countries; then list, view, edit, activate, suspend, and soft-delete it (Global Req: "Manage installation companies within assigned country or countries"; RBAC "Installer Management - Full Access (own data only)").

Architectural Approach

Creation runs in a single MySQL transaction (installer + primary-admin account + module rows), generates a unique installer_id under the distributor, then enqueues an activation email and writes an audit entry. The selectable module set is constrained server-side to the distributor's HQ-allowed set (the funnel). An installer cannot be suspended/deleted while it has active properties.

Persistence

MySQL installer (installer_id PK, distributor_id FK, country, modules[], status) + primary-admin account.

Module 06

Module Enablement (Funnel)

Enable a subset of HQ-permitted modules per installer

Role: Distributor (Web Portal)
NestJS module: ModuleEnablementModule
Services: FunnelService

Objective

Enable/disable modules for each installer, restricted to the HQ-allowed subset (Global Req: "Select and enable modules for installation companies (within HQ-defined scope)"; RBAC "Modules Configuration - Enable subset for Installers").

Architectural Approach

The distributor's allowed set (the funnel ceiling defined by HQ) bounds what can be enabled per installer. On change, the backend refreshes the Redis module-flag cache so the funnel is enforced API-level (ModuleEnabledGuard) on the next request - technically enforced at the API level, not just the UI (Global Req). An installer can never receive more than the distributor was permitted.

Scope

Per-installer module enablementSubset of HQ-allowed setAPI-enforced funnelRedis flag propagation

Endpoints & Rules

GET/PUT /api/v1/installers/{id}/modules (subset of distributor's HQ-allowed set; else 403). Save refreshes module flags in Redis; change audited.

Persistence

MySQL installer_module (enablement rows); flags cached in Redis.

Module 11

Escalations & Temporary Access

Handle installer escalations; forward to HQ; time-limited access

Role: Distributor (Web Portal)
NestJS module: EscalationModule
Services: EscalationService · TemporaryAccessService

Objective

Receive and resolve escalations raised by installers, or forward them to HQ when out of scope (Global Req escalation path: User -> Property -> Installer -> Distributor -> HQ; direct User -> HQ not permitted). Also governs temporary, time-limited access grants between tiers for issue resolution.

Architectural Approach

Escalations follow the fixed path; the Distributor resolves within scope or forwards to HQ with status tracking. Per Global Req, when issue resolution requires it, temporary access may be explicitly granted by the lower tier on a time-limited basis, and HQ may be granted temporary read-only access to distributor-scope configuration - such access is requested, time-limited, fully audited, and auto-revoked on expiry.

Scope

Receive installer escalationsResolve within scopeForward to HQ (out of scope)Time-limited temporary access (audited)

Escalation Flow

1
Installer raises escalation
Per path: User -> Property -> Installer -> Distributor.
GET /api/v1/escalations (scoped queue).
2
Resolve within scope
Distributor handles and updates status; installer notified.
PUT /api/v1/escalations/{id} (resolve / update).
3
Forward to HQ (if out of scope)
Escalated up the path; status tracked.
POST /api/v1/escalations/{id}/forward -> HQ. May involve time-limited, audited temporary access.

Endpoints & Rules

GET/PUT /api/v1/escalations · POST /api/v1/escalations/{id}/forward. Temporary access grants are read-only, time-limited, fully audited, auto-revoked on expiry.

Persistence

MySQL escalation (scoped); temporary-access grants audited -> Cosmos System Config trail.
