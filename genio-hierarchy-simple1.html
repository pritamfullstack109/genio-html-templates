Architecture & Cross-Cutting Standards

Technical Architecture Overview

The contracts every module inherits

The HQ Portal is one role-rendering of the global React Admin SPA. All requests traverse Azure API Management -> the AKS cluster (NestJS) -> data stores. The backend is the single point of authorization: it derives the caller's tier and scope from the JWT on every request and enforces RBAC and the funnel server-side. Unlike downstream tiers, the Global Admin's scope is system-wide (no installer_id / condo_id restriction) but read access to operational data is view-only (oversight, not operation).

REQUEST PATH HQ Global Admin (React SPA) --HTTPS + Bearer JWT--> Azure API Management (routes by cluster_id) | v NestJS API Gateway / BFF ---> Guard chain: JwtAuthGuard -> SessionGuard -> RolesGuard (role=GLOBAL_ADMIN, tier=L1) -> ScopeGuard (system-wide) -> ModuleEnabledGuard | (AuditInterceptor logs every config write) v Domain services (Distributor / Cluster / Role / Policy / Licence / Audit / ...) ---> Data stores GLOBAL SHARED SERVICES Shared Auth DB centralized login / JWT issuance / user registry (resolves cluster_id, distributor_id, role) Azure API Mgmt single gateway; routes each request to the correct regional cluster by cluster_id Static Web Apps one React Admin Portal (HQ/Distributor/Installer) + one Condo Management web app; CDN-distributed Azure Key Vault all secrets, certificates, encryption keys - by alias MeetMe Licensing global shared SIP intercom licence pool DATA STORES MySQL (Azure) distributors (entity_id), clusters, module catalogue, roles, policies, licences, HQ staff Cosmos DB System Configuration Audit Trail (append-only, immutable, per-region TTL) Redis sessions · module-flag (funnel) cache · perms cache Azure Blob report exports / audit archives (retention mirrors the DB) Regional Clusters Europe / Asia / Middle East - fully isolated Azure Resource Groups; data never crosses regions

Cross-Cutting Standards (every module inherits)

CC-01 Funnel (top) - HQ defines the distributor module ceiling, enforced API-level via ModuleEnabledGuard + Redis flags  ·  CC-02 Scope - ScopeGuard resolves HQ to system-wide (aggregate, view-only on ops)  ·  CC-03 Soft delete - status (reactivatable; hard purge only via retention job)  ·  CC-04 Encryption - PII columns + Key Vault  ·  CC-05 Validation - ValidationPipe + DTOs  ·  CC-06 Audit - AuditInterceptor -> System Config Audit Trail (all HQ config writes)  ·  CC-07 Onboarding email - async queue  ·  CC-08 Data residency - per-cluster isolation; EU = GDPR.

Standard Error Codes

400 validation · 401 unauthenticated · 403 out-of-scope / forbidden / funnel · 404 not found · 409 conflict (e.g. duplicate email) · 422 semantically invalid · 429 rate-limit · 500 internal. Every response uses one envelope with a correlation_id.

Platform Context

Why the HQ Tier is Net-New

How the Global Admin tier fits the Geo-Partitioned Hierarchical Multi-Tenant platform

The legacy iPlusLiving system (PHP/Yii2 + AWS) had no HQ / governance tier - it was scoped by company_id / condo_id only. Genio360 introduces the four-tier funnel (HQ -> Distributor -> Installer -> Property) as a fresh rebuild on Azure. Every HQ capability in this document is therefore NEW; there is no As-Is baseline to migrate.

The Four Tiers (Ownership-Based Access)

L1 HQ / FAP - governance, configuration, oversight (this document).  L2 Distributor - channel layer, manages installers within a country (own entity_id).  L3 Installer - execution layer, onboards properties (own installer_id).  L4 Property Manager - site operations. Each layer acts only on what it owns; upstream may observe downstream but not operate it; downstream has no upward/lateral visibility.

Open Items (to confirm with FAP)

OI-1 Entity vs Region: Global Requirements uses "Entity" inconsistently. This SRS adopts cluster_id = region (infra + retention grouping), entity_id = distributor (L2).  OI-2 Escalated access mechanics to be confirmed in discovery.  OI-3 cluster infrastructure provisioning is DevOps (precedes onboarding).  OI-4 final role taxonomy / 66-module catalogue / funnel mapping.  OI-5 MeetMe allocation path (via Distributor vs direct to property/unit).

Scope Note - Excluded from this revision

The following were present in earlier reference material but are not stated in Global Requirements r5.0, so they are excluded here pending a separate source (SOW / NFR) sign-off: Developers / API Access, Localization (en/zh-SG/vi), and MFA / OTP plus NFR-only login detail (single-session, lockout, forgot-password). Authentication retains only what Global Requirements states: Shared Auth DB credential validation + JWT issuance.

Module 01

Authentication

Shared Auth DB credential validation + JWT issuance

Role: HQ / Global Admin (Web Portal)
NestJS module: AuthModule
Services: AuthService · TokenService

Objective

Authenticate the Global Admin against the global Shared Auth DB and issue the role-and-scope JWT that every other module relies on for authorization. (Per Global Requirements: the Shared Auth DB manages login credentials and JWT token generation; regional assignment - cluster_id, distributor_id, role - is resolved at login and embedded in the token.)

Architectural Approach

Credential validation is delegated to the Shared Auth DB; on success a signed JWT is issued carrying the user's role + scope. Azure API Management routes the request to the correct regional cluster by cluster_id; the NestJS backend re-reads role/ids from the JWT on every request and enforces RBAC server-side. For HQ the token resolves to tier=L1, system-wide scope.

Scope

Credential validation (Shared Auth DB)JWT issuanceScope resolution (cluster_id / role)Server-side RBAC render

Login Flow

1
Submit credentials
React posts Email/Password over HTTPS.
POST /api/v1/auth/login -> validated against the Shared Auth DB. Invalid -> 401.
2
JWT issued
Signed token carries role + scope.
JWT: sub, role=GLOBAL_ADMIN, tier=L1, cluster_id=ALL. APIM then routes by cluster_id.
3
RBAC render
ProtectedRoute renders only Global-Admin routes; the backend re-validates the JWT on every request and stays authoritative.

Endpoints & Rules

POST /api/v1/auth/login -> Shared Auth DB validation, JWT issuance. Backend re-reads role/ids from the JWT on every request (single point of authorization). All authentication events are written to the audit trail (Login History).

Persistence

Credentials/registry in the Shared Auth DB; session context in Redis.

Module 04

HQ Staff & Internal Roles

Create internal FAP staff accounts and assign role-scoped access

Role: HQ / Global Admin (Web Portal)
NestJS module: HqStaffModule
Services: HqStaffService

Objective

Create and manage HQ internal staff accounts - the Fermax Asia Pacific employees who operate the HQ portal alongside the primary Global Admin - and assign each a role. (Per Global Req Process Flow: "HQ creates a staff account" precedes region provisioning.) Staff are internal employees, distinct from Distributors (which are external downstream entities, see Module 05).

Architectural Approach

Scoped CRUD over HQ staff. Roles are defined in Global Roles (Module 07); this module only assigns an existing L1 role to a staff member. New staff receive an activation email (async queue); access is constrained to HQ-tier capabilities only.

Access Model - full vs limited (role-driven)

A staff member's access is exactly what their assigned role permits - it can be full or limited. The primary Global Admin has full access; other staff get scoped roles (least-privilege). Example: a staff with an "Onboarding Manager" role (Distributor Management = allowed) can create distributors; a staff with an "Auditor" / "Viewer" role cannot - they only read. This delegates work, enforces least-privilege, and keeps every action attributable in the audit trail.

Scope

CRUD HQ staff (internal employees)Assign an existing L1 roleFull or limited access (role-driven)Activation emailSoft delete

Endpoints & Rules

GET/POST/PUT/DELETE /api/v1/hq/staff (paginated/search; email unique) · PUT /api/v1/hq/staff/{id}/role assigns an L1 role defined in Module 07. A staff can perform an action (e.g. create a distributor) only if their role grants that permission. Delete soft; all actions audited (Role Changes / Permission Changes).

Persistence

MySQL hq_staff_user (PII encrypted); role FK -> role (tier=L1, defined in Module 07).

Module 05

Distributor Management

Create / Activate / Suspend / Soft-delete / List - the core governance entity

Role: HQ / Global Admin (Web Portal)
NestJS module: DistributorModule
Services: DistributorService · DistributorOnboardingService

Objective

Create a Distributor company (Distributor Entity) with its primary admin and allowed modules, assigned to a region cluster; then list, view, edit, activate, suspend, and soft-delete it - the top of the onboarding funnel (Global Req: "Create and Manage Distributors").

Architectural Approach

Creation runs in a single MySQL transaction (distributor + primary-admin account + allowed-module rows), generates a unique entity_id within the cluster, then enqueues an activation email and writes an audit entry. Allowed modules are constrained server-side to the global catalogue (the funnel). A distributor cannot be suspended/deleted while it has active installers.

Module 06

Module Catalogue & Funnel

Define the global catalogue and each distributor's permitted subset

Role: HQ / Global Admin (Web Portal)
NestJS module: ModuleCatalogueModule
Services: CatalogueService · FunnelService

Objective

Maintain the global module catalogue (the 66-module master set) and define, per distributor, the subset they may use - the top of the progressive-enablement funnel (Global Req: "Module Configuration / funnel effect").

Architectural Approach

The catalogue is the master list; per-distributor enablement rows define the downstream ceiling. On change, the backend refreshes the Redis module-flag cache per cluster so the funnel is enforced API-level (ModuleEnabledGuard) on the next request - technically enforced at the API level, not just the UI (Global Req). A distributor can never enable more than its parent (HQ) permitted.

Scope

CRUD global cataloguePer-distributor allowed subsetFunnel ceiling (API-enforced)Redis flag propagation

Endpoints & Rules

GET/POST/PUT /api/v1/modules/catalogue · GET/PUT /api/v1/distributors/{id}/allowed-modules (subset of catalogue; else 422). Save evicts/refreshes moduleFlags:{cluster_id} in Redis; change audited.

Persistence

MySQL module_catalogue, distributor_allowed_module; flags cached in Redis per cluster.

Module 09

Module 11

Escalated Temporary Access NEW MODULE

Time-limited, read-only, audited, auto-revoked

Role: HQ / Global Admin (Web Portal)
NestJS module: EscalatedAccessModule
Services: EscalatedAccessService · AccessExpiryWorker

Objective

Where a Distributor cannot resolve a technical issue affecting an Installer, HQ may be granted temporary, explicitly-authorized, read-only access to the relevant configuration (Global Req exception to "oversight without operational control").

Architectural Approach

Access must be requested by the lower tier and is granted time-bound and read-only. A background worker auto-revokes on session expiry. Every grant, use, and revocation is fully logged in the audit trail. OI-2: mechanics to be confirmed in discovery.
